# Personal Knowledge Assistant

A local, privacy-first RAG (Retrieval-Augmented Generation) agent that answers questions from your own documents — resumes, notes, PDFs, markdown files — using locally-hosted LLMs and embedding models via [Ollama](https://ollama.com), [LangChain](https://www.langchain.com/), and [Chroma](https://www.trychroma.com/).

Everything runs on your machine. No documents or queries leave your computer.

## Features

- **Fully local** — LLM inference and embeddings run through Ollama; no external API calls required.
- **Pluggable models** — swap embedding models and LLMs via a Registry/Factory pattern without touching core logic.
- **Multi-format ingestion** — loads `.md`, `.txt`, and `.pdf` files from a documents folder.
- **Persistent vector store** — Chroma-backed, with a separate collection per embedding model (so you can compare models without cross-contamination).
- **Source attribution** — every answer lists which document(s) it was grounded in.
- **Agent-based architecture** — built on LangChain's `create_agent` with a custom retrieval middleware, rather than a hardcoded RAG chain.

## Architecture

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────┐
│  main.py    │────▶│  rag.py           │────▶│  Chroma DB  │
│ (entrypoint)│     │ (load, chunk,     │     │ (per-model  │
└──────┬──────┘     │  embed, persist)  │     │  collection)│
       │            └──────────────────┘     └─────────────┘
       ▼
┌─────────────┐     ┌──────────────────┐     ┌─────────────┐
│  agent.py   │────▶│  providers.py     │────▶│   Ollama    │
│ (agent +    │     │ (registry: LLM &  │     │  (local     │
│  middleware)│     │  embedding loaders)│     │   models)   │
└─────────────┘     └──────────────────┘     └─────────────┘
```

**Key design decisions:**

- **Registry pattern for models** (`registry.py`, `providers.py`) — adding a new LLM or embedding provider means registering a new factory function, not editing existing dispatch logic. Imports are lazy (deferred until a provider is actually used), so you don't pay the cost of optional heavy dependencies (e.g. `torch` for HuggingFace models) unless you use them.
- **Enum-based model identifiers** (`models.py`) — `EmbeddingModel` and `LLMModel` enums give validated, autocomplete-friendly model selection instead of magic strings.
- **Retrieval as middleware, not a tool** (`agent.py`) — a custom `AgentMiddleware` intercepts before each model call, retrieves relevant chunks, and injects them as a system message. This keeps retrieval deterministic and separate from the LLM's own tool-calling behavior.
- **Per-embedding-model vector stores** — since different embedding models produce incompatible vector spaces, each gets its own Chroma persist directory, keyed by model name.

## Project Structure

```
personal_knowledge_assistant/
├── build.py            # One-time environment setup (dirs, .env, poetry install)
├── config.py            # Paths, chunking, retrieval, and prompt constants
├── models.py             # EmbeddingModel / LLMModel enums
├── registry.py            # Generic key → factory registry
├── providers.py             # Registered loaders for embeddings and LLMs
├── rag.py                    # Document loading, chunking, vectorstore build/load
├── agent.py                    # Agent state, retrieval middleware, agent builder
├── main.py                      # CLI entrypoint / chat loop
├── pyproject.toml                 # Poetry dependencies
├── data/
│   ├── documents/                   # Drop your source files here
│   └── vector_db/
│       └── <embedding-model>/          # One Chroma store per embedding model
└── .env                                  # PROJECT_ROOT (generated by build.py)
```

## Prerequisites

- **Python** 3.10–3.13 (3.14 is not yet supported — several dependencies, including `torch` and `triton`, don't have compatible wheels/version ceilings for 3.14 at time of writing)
- **[Poetry](https://python-poetry.org/)** for dependency management
- **[Ollama](https://ollama.com)** installed and running locally

## Setup

**1. Install Ollama and pull the models you plan to use:**

```bash
ollama pull nomic-embed-text
ollama pull llama3.1        # or any other chat model you plan to use
```

**2. Run the setup script:**

```bash
python3 build.py
```

This will:
- Create `data/documents/`, `data/vector_db/`, and `src/`
- Write `PROJECT_ROOT` into a `.env` file
- Configure Poetry to use an in-project `.venv`
- Lock and install all dependencies

**3. Add your documents:**

Drop `.md`, `.txt`, or `.pdf` files into `data/documents/`.

**4. Run the assistant:**

```bash
poetry run python main.py
```

## Usage

```
You: What was my role at Western Digital?
Answer: ...

Sources:
- data/documents/resume.pdf
```

Type `exit` or press Ctrl+C to quit.

## Configuration

Key settings live in `config.py`:

| Setting | Purpose |
|---|---|
| `CHUNK_SIZE` / `CHUNK_OVERLAP` | Controls how documents are split before embedding |
| `RETRIEVAL_K` | Number of chunks retrieved per query |
| `SYSTEM_PROMPT` | Base instruction given to the LLM alongside retrieved context |

To add a new model:

```python
# models.py
class EmbeddingModel(str, Enum):
    ...
    MY_NEW_MODEL = "some-model-name"
```

```python
# providers.py
@embedding_registry.register(EmbeddingModel.MY_NEW_MODEL)
def _load_my_new_model(model: EmbeddingModel):
    from some_package import SomeEmbeddings
    return SomeEmbeddings(model=model.value)
```

No other files need to change.

## Rebuilding the vector store

Each embedding model's vector store is only built once and reused on subsequent runs. To force a rebuild (e.g. after adding documents, or changing `CHUNK_SIZE`/`CHUNK_OVERLAP`):

```bash
rm -rf data/vector_db/<embedding-model-name>
```

## Known limitations

- Retrieval is pure vector similarity — no re-ranking, hybrid search, or metadata filtering yet. Ambiguous queries spanning multiple similar documents (e.g. several past jobs) may retrieve the wrong chunk.
- Large document sets are embedded in batches to avoid overwhelming local Ollama runners; very large corpora may take a while to index on first run.
- No conversation memory across turns — each question is answered independently.

## Roadmap ideas

- Conversation history / multi-turn context
